# Check — Mobile binary protections (release-build hardening)

**What you're looking for:** mobile apps that ship to the store as a wide-open, easy-to-read binary — no code shrinking or obfuscation, `debuggable` left on in the release build, debug symbols and readable strings baked into the shipped artifact, and (for apps that handle real money or sensitive data) zero root/jailbreak or app-integrity checks. None of these is a "your app is hacked" emergency on its own. They're **defense in depth**: each one makes it cheaper and faster for someone to download your published app, pull it apart, read your logic, find the soft spots, and either clone it or build a tampered version. This is the **medium / advisory** end of the mobile checklist — you flag it after the real leaks (keys, cleartext, insecure storage) are handled, and you flag it gently.

This is a config check, not a binary check. **We do not decompile or analyze a live `.ipa`/`.apk`** — Lictor reads the repo and the build configuration that *decides* how the shipping binary gets built. If the config says "ship it raw," that's the finding. It applies whether the app was hand-built in Xcode/Android Studio or generated by Flutter, React Native, Expo, Capacitor, or a "make me an app" tool — they all funnel through the same Gradle / Xcode / packaging settings.

> **Companion check:** `mobile.md` covers the things that *leak data* — baked-in server keys, plain-HTTP traffic, tokens in plaintext storage, exported components, secrets in logs. This module is the separate layer: even with all of that clean, is the *shipping binary itself* hardened? Run `mobile.md` first; this is the level-up.

## How to scan

You're reading build config, not the binary. The interesting files are the Android Gradle build scripts, the iOS project/build settings, and the framework packaging config.

```bash
# ── ANDROID: release build that does NOT shrink/obfuscate ─────────────
# Find the release buildType block and see whether minify/shrink/proguard is on
grep -rEn --include='*.gradle' --include='*.gradle.kts' --exclude-dir={node_modules,build} \
  -E 'minifyEnabled|isMinifyEnabled|shrinkResources|isShrinkResources|proguardFiles|isDebuggable|debuggable' \
  . 2>/dev/null

# Strong red flag: release explicitly turns shrinking OFF
grep -rEn --include='*.gradle' --include='*.gradle.kts' --exclude-dir={node_modules,build} \
  -E '(minifyEnabled|isMinifyEnabled)\s*=?\s*false' \
  . 2>/dev/null

# Strong red flag: a release variant left debuggable
grep -rEn --include='*.gradle' --include='*.gradle.kts' --exclude-dir={node_modules,build} \
  -E '(isDebuggable|debuggable)\s*=?\s*true' \
  . 2>/dev/null

# See the whole release block in context (read these by hand)
grep -rEn --include='*.gradle' --include='*.gradle.kts' --exclude-dir={node_modules,build} \
  -A 12 -E 'release\s*\{|getByName\("release"\)|create\("release"\)' \
  . 2>/dev/null

# ── iOS: debug symbols / no stripping in the Release config ───────────
# Look at build settings in the pbxproj or an .xcconfig
grep -rEn --include='*.pbxproj' --include='*.xcconfig' \
  -E 'STRIP_INSTALLED_PRODUCT|DEPLOYMENT_POSTPROCESSING|GCC_OPTIMIZATION_LEVEL|SWIFT_OPTIMIZATION_LEVEL|DEBUG_INFORMATION_FORMAT|COPY_PHASE_STRIP|ONLY_ACTIVE_ARCH' \
  . 2>/dev/null

# A DEBUG / -Onone optimization level leaking into a Release build is the tell
grep -rEn --include='*.xcconfig' --include='*.pbxproj' \
  -E 'SWIFT_OPTIMIZATION_LEVEL\s*=?\s*"?-Onone|GCC_OPTIMIZATION_LEVEL\s*=?\s*0|SWIFT_ACTIVE_COMPILATION_CONDITIONS.*DEBUG' \
  . 2>/dev/null

# ── FLUTTER / RN / EXPO: confirm release builds aren't profile/debug ──
# Flutter: the android release block still controls shrinking (above).
# RN/Expo: same Gradle file under android/app/build.gradle; plus check easjson build profiles
grep -rEn --exclude-dir=node_modules \
  -E '"developmentClient"\s*:\s*true|"buildType"\s*:\s*"apk"|gradleCommand.*assembleDebug' \
  eas.json 2>/dev/null

# ── ROOT / JAILBREAK / INTEGRITY checks present at all? ───────────────
# (Absence is only advisory, and only for sensitive apps — see guards)
grep -rEn --exclude-dir={node_modules,Pods,build,.git,.dart_tool} \
  --include='*.swift' --include='*.kt' --include='*.java' --include='*.dart' \
  --include='*.js' --include='*.ts' --include='*.gradle' \
  -iE 'jailbreak|jailbroken|isRooted|rootbeer|RootBeer|SafetyNet|PlayIntegrity|DeviceCheck|AppAttest|freerasp|jail_monkey|flutter_jailbreak|emulator.*detect|isEmulator|frida|debugger.*detect' \
  . 2>/dev/null
```

Also worth opening by hand:
- **`android/app/build.gradle`** (or `build.gradle.kts`) — the `buildTypes { release { ... } }` block. This is where 90% of findings live.
- **`*.pbxproj`** and any **`Release.xcconfig`** — the iOS Release build settings.
- **`proguard-rules.pro`** — present and referenced? (its existence implies R8/ProGuard is wired up).
- **`eas.json`** (Expo) — the build profiles; make sure the store profile isn't a dev/debug client.

## The dangerous patterns

### Pattern 1 — Android release with no code shrink / obfuscation (MEDIUM)

```groovy
// android/app/build.gradle — the AI-default that ships everything raw
android {
  buildTypes {
    release {
      minifyEnabled false          // ← no R8/ProGuard: full class & method names,
      shrinkResources false        //    readable strings, dead code all shipped
      // (no proguardFiles line at all)
    }
  }
}
```
```kotlin
// Kotlin DSL flavour — same problem
buildTypes {
    getByName("release") {
        isMinifyEnabled = false
        isShrinkResources = false
    }
}
```

With `minifyEnabled false`, R8/ProGuard never runs, so your release `.apk`/`.aab` ships with **original class and method names, all your string literals, and unused code intact**. Anyone can `apktool` / `jadx` the published app and read your business logic almost like source — which endpoints you call, how you compute prices or unlock premium features, where the soft spots are. Turning R8 on renames everything to `a`, `b`, `c`, strips dead code, and makes the app meaningfully harder (and slower) to reverse-engineer and clone. This is the single most common mobile-hardening miss, because the Android template historically shipped with `minifyEnabled false` and AI scaffolds copy it.

**MEDIUM / advisory.** It's not a data leak by itself — but for an app with any proprietary logic, paid features, or anti-fraud rules, raw bytecode is a gift to a cloner.

### Pattern 2 — A release variant left `debuggable` (MEDIUM → HIGH if it reaches the store)

```groovy
buildTypes {
  release {
    debuggable true              // ← release build is debuggable
  }
}
```
```kotlin
getByName("release") { isDebuggable = true }
```

A `debuggable` release lets anyone attach a debugger (jdwp), dump memory, read variables live, and hook the running app. It also weakens other OS protections. The usual cause: a dev flipped it on to debug a release-only crash and forgot to flip it back, or copied a build flavour. **MEDIUM** as a config smell; treat it as **HIGH** if you can see it actually flows into the store build (e.g. it's on the default `release` type with no separate "staging" flavour). The store/play console will often reject a `debuggable` upload — but don't rely on that; remove it.

### Pattern 3 — Debug symbols / readable strings left in the shipped artifact (MEDIUM)

```groovy
// Android: NOT stripping native debug symbols from the release bundle
android {
  // (no `ndk { debugSymbolLevel ... }` / no symbol upload, OR explicitly:)
  buildTypes { release { ndk { debugSymbolLevel 'FULL' } } } // shipping full symbols in the artifact
}
```
```
// iOS Release config that behaves like a debug build:
SWIFT_OPTIMIZATION_LEVEL = -Onone     // ← no optimization in Release
GCC_OPTIMIZATION_LEVEL  = 0           // ← same, for ObjC/C
STRIP_INSTALLED_PRODUCT = NO          // ← symbols not stripped from the binary
DEPLOYMENT_POSTPROCESSING = NO        // ← strip phase skipped
SWIFT_ACTIVE_COMPILATION_CONDITIONS = DEBUG   // ← DEBUG code compiled into Release
```

Unstripped binaries and `-Onone`/`DEBUG` Release builds keep function names, type metadata, assertion strings, and file paths inside the shipped binary, so a `strings` / `class-dump` / `nm` pass reads your structure for free. The fix is to let the Release config do its job: `-O` / `-Os` optimization, `STRIP_INSTALLED_PRODUCT = YES`, `DEBUG_INFORMATION_FORMAT = dwarf-with-dsym` (keep the dSYM for *your* crash symbolication, but ship a stripped binary), and on Android upload native debug symbols to Play separately rather than baking them into the artifact. Crash symbolication still works because you keep the dSYM/symbol file on your side — you just don't hand it to everyone who downloads the app.

**MEDIUM / advisory.** Strongest signal is a Release config that is literally a Debug config in disguise (the `-Onone` + `DEBUG` flag combination above).

### Pattern 4 — No root / jailbreak / integrity check at all, on a high-value app (MEDIUM, advisory only)

```kotlin
// A banking / wallet / health app with zero device-integrity posture:
//   no RootBeer / Play Integrity / SafetyNet (Android)
//   no jailbreak check / DeviceCheck / App Attest (iOS)
//   no freerasp / jail_monkey / flutter_jailbreak_detection (RN/Flutter)
// → grep for those names came back completely empty
```

On a rooted or jailbroken device, the OS sandbox guarantees you'd normally lean on (secure storage isolation, anti-tamper, screen-overlay protection) are weaker, and tools like Frida can hook your app live. For an app that moves money, stores health records, gates a paid product, or enforces anti-cheat, a **root/jailbreak signal plus a server-side app-integrity attestation** (Play Integrity API / Apple App Attest + DeviceCheck) lets you treat those sessions with extra suspicion — step up auth, refuse high-risk actions, or flag for review. The complete *absence* of any such check on a sensitive app is worth a gentle MEDIUM "consider adding."

**This is the easiest pattern to false-positive, so flag it narrowly.** Only raise it when the app clearly handles high-value data (you can see payments, banking, health, identity, or a paid-content gate in the repo). For a to-do list, a recipe app, a marketing app, or a content reader — **do not flag missing root detection at all.** It's pure noise there, and over-flagging it trains founders to ignore the report.

## Safe patterns

**Android release that shrinks + obfuscates and is not debuggable:**
```groovy
buildTypes {
  release {
    minifyEnabled true              // R8/ProGuard runs
    shrinkResources true            // unused resources stripped
    proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'),
                  'proguard-rules.pro'
    // debuggable defaults to false — leave it that way
    ndk { debugSymbolLevel 'SYMBOL_TABLE' }   // or upload symbols to Play, not bake them in
  }
}
```

**iOS Release config that strips and optimizes (keep the dSYM for your own symbolication):**
```
// Release.xcconfig
SWIFT_OPTIMIZATION_LEVEL = -O
GCC_OPTIMIZATION_LEVEL   = s
STRIP_INSTALLED_PRODUCT  = YES
DEPLOYMENT_POSTPROCESSING = YES
DEBUG_INFORMATION_FORMAT = dwarf-with-dsym     // dSYM stays with you for crash reports
// no DEBUG in SWIFT_ACTIVE_COMPILATION_CONDITIONS for Release
```

**Integrity posture for a sensitive app (server-verified, not client-only):**
```kotlin
// Android — Play Integrity verdict checked on YOUR server before high-risk actions
val token = IntegrityManagerFactory.create(ctx)
    .requestIntegrityToken(/* nonce from your server */)
// → send token to your backend, verify there, decide whether to allow the action
```
```swift
// iOS — App Attest, again verified server-side
let attestation = try await DCAppAttestService.shared.attestKey(keyId, clientDataHash: hash)
// → backend validates with Apple before trusting the device
```
The key word is **server-verified**: a client-only `isJailbroken()` that just shows a warning is trivially patched out of an obfuscated-off binary. Make the decision on your backend off an attestation token.

## Report a finding as

**Title:** "Your Android app ships unprotected — anyone can unzip the published app and read your code"

(adapt per pattern: "Your release build is debuggable — someone can attach a debugger to the published app", "Your iOS Release build ships with debug symbols, so your code structure is readable", "Your [banking/wallet/health] app has no root/jailbreak or integrity check")

**Detail:**
> `android/app/build.gradle:34` sets `minifyEnabled false` (and there's no `proguardFiles` line) inside the `release { }` block, so R8/ProGuard never runs on your shipping build. Here's why that matters: an `.aab`/`.apk` is just a zip, and with no shrinking or obfuscation it ships with your **original class names, method names, and string literals intact**. Anyone can download your published app and run `jadx` on it to read your logic almost like source — which API calls you make, how you decide who gets premium, where your anti-fraud checks are. None of your *secrets* leak from this alone (that's a separate check), but it hands a would-be cloner or attacker a free map of how your app works.
>
> This is **advisory / defense-in-depth**, not an emergency — it's the kind of thing worth doing before you have a popular app worth copying, not something that's actively exposing user data tonight.
>
> **What to do (a 5-minute change):**
> 1. Turn on shrinking + obfuscation for release in `android/app/build.gradle`:
>    ```groovy
>    buildTypes {
>      release {
>        minifyEnabled true
>        shrinkResources true
>        proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'),
>                      'proguard-rules.pro'
>      }
>    }
>    ```
> 2. Build the release variant and run your app once — if anything breaks (usually reflection / a JSON library needing keep rules), add the needed `-keep` lines to `proguard-rules.pro`. R8 prints exactly what to keep.
> 3. While you're in that block, confirm there's no `debuggable true` on the release type, and that `debuggable` isn't being copied in from another flavour.
> 4. Sanity check after build: `unzip -p app-release.aab | strings | head` should look like noise (`a`, `b`, `c` class names), not your readable code.
>
> If this is intentionally out of scope — an internal-only build, an open-source app where the code is public anyway, or an early prototype — that's a legitimate call; note it and move on.

Repeat the report block for each pattern you found, swapping in the right story:
- **Debuggable release:** "Your `release` build sets `debuggable true`, so anyone can attach a debugger to the published app, read variables live, and hook it. Remove `debuggable true` from the release buildType (it should default to false)." Treat as HIGH if it clearly reaches the store build.
- **iOS debug symbols / `-Onone` in Release:** "Your iOS Release config builds like a Debug build (`SWIFT_OPTIMIZATION_LEVEL = -Onone`, `STRIP_INSTALLED_PRODUCT = NO`, `DEBUG` compilation flag), so the shipped binary keeps function names and structure that a `strings`/`class-dump` pass reads off easily. Set `-O`, `STRIP_INSTALLED_PRODUCT = YES`, drop `DEBUG` from the Release conditions, and keep the dSYM on your side for crash reports."
- **No integrity check on a sensitive app:** "Your app handles [payments/health/identity] but has no root/jailbreak signal and no server-verified integrity attestation (Play Integrity / App Attest). On a rooted or jailbroken phone your protections are weaker and the app can be hooked with Frida. Consider adding an attestation check that your *backend* verifies before allowing high-risk actions." Advisory only — never raise this for a non-sensitive app.

## What NOT to flag (false-positive guards — read this before reporting)

This check is **medium/advisory by nature** and easy to over-fire. Defense-in-depth is good to have, but crying wolf on it buries the real findings. Don't flag:

- **Anything in the `debug` buildType.** `minifyEnabled false`, `debuggable true`, `-Onone`, `DEBUG` flags — these are *correct and expected* in the debug/dev build. Confirm you're reading the **`release`** (or the store/distribution) variant before flagging. A finding here requires the bad setting to be in the *shipping* config.
- **Apps where obfuscation is intentionally out of scope.** Open-source apps (the code is already public — obfuscation buys nothing), internal/enterprise distribution, school projects, prototypes, and explicitly-OSS repos. If there's a LICENSE that publishes the source, don't flag "no obfuscation."
- **Missing root/jailbreak detection on a non-sensitive app.** A recipe app, a to-do list, a blog reader, a marketing landing app — these have nothing for an attacker to gain from a rooted device. Do **not** flag missing integrity checks here. Only raise it when you can actually see high-value data (payments, banking, health, identity, paid-content gate) in the repo.
- **Cert pinning, anti-tamper, or anti-Frida as a hard requirement.** These are advanced hardening, appropriate for finance/health, and counterproductive (false sense of security, support pain) on a normal app. Mention them only as optional LOW/INFO level-ups for sensitive apps, never as a finding on a typical app.
- **A `release` block that's already hardened.** `minifyEnabled true` + `proguardFiles` present, or an iOS Release with `-O` + `STRIP_INSTALLED_PRODUCT = YES`, means the app is doing the right thing. Seeing a `proguard-rules.pro` file or `freerasp`/`RootBeer`/`PlayIntegrity`/`AppAttest` in the deps is the *good* sign — don't flag the presence of hardening.
- **Frameworks that handle this for you.** A bare Flutter/RN/Expo release built via the standard `flutter build appbundle` / `eas build --profile production` path may already apply the platform defaults; check the actual Gradle/xcconfig before assuming it's unhardened. Don't flag a framework's managed release just because you didn't see a hand-written `release {}` block — confirm what the build actually does.
- **`debuggable`/`-Onone` behind a build flag or a non-release flavour.** A `staging`/`internal` flavour that's debuggable is a legitimate internal build, not the store artifact. Only the variant that ships to users counts.
- **Keeping a dSYM / mapping file.** Uploading a dSYM (iOS) or `mapping.txt` (Android R8) to your crash reporter is *correct* — that's how you symbolicate crashes after obfuscation. It is not "shipping debug symbols to users"; it stays on your side. Don't confuse the two.

When in doubt, the deciding question is: **"is this the *shipping* build config, and does the app have something worth protecting (proprietary logic, paid features, money, or sensitive data)?"** If yes, and the release ships raw/debuggable/unstripped → flag it as a MEDIUM, advisory, "do this before you're big" item. If it's a debug build, an open-source/prototype app, or a non-sensitive app missing root detection → don't flag it; at most a one-line INFO heads-up.

🟡 Medium

No Content-Security-Policy header set

all routes

The site doesn't set a CSP header. For a marketing site with no user-uploaded content this is low-risk, but a strict CSP would prevent the entire class of injected-script attacks if a future page ever includes user-controllable content (a contact form, a search box, a comment thread).

What can go wrong: if a future content addition trusts user input without realizing, a single XSS bug becomes a much bigger problem than it needs to be. CSP is the safety net for the mistake we haven't made yet.

How to fix (5 min)

Add to landing/_headers (Cloudflare Pages syntax):

/*
  Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' fonts.googleapis.com; font-src fonts.gstatic.com; img-src 'self' data:; frame-ancestors 'none'
🟡 Medium

No Permissions-Policy header restricting features

all routes

The site doesn't restrict access to high-risk browser features (camera, microphone, geolocation, payment). Same shape as the CSP finding — defense in depth against a future page that doesn't yet exist.

How to fix (3 min)

Add to landing/_headers:

/*
  Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=(), usb=()
⚪ Info

Google Fonts loaded from fonts.googleapis.com

all routes

The site loads Cormorant Garamond, Inter, and JetBrains Mono from Google Fonts. This is a third-party request from every page load — a privacy and performance tradeoff, not a security finding. Self-hosting the fonts would eliminate the request entirely.

Not flagging this as a problem — just letting you know it's there. The current loadout includes the preconnect hint, which is the right move if Google Fonts is staying.

# Session memory — 2026-05-21 (the v3 origin night)

This is the canonical session log every v3 agent should read on first boot. It explains WHY v3 exists, WHAT was learned, and HOW to act on the lessons.

## Timeline

### Morning (before this session)
- Raffa had been running v0.2 (just shipped that morning) against bounty corpus + security vendors
- Had submitted ~12 reports across H1/BC, several already closed Spam or Not Applicable
- HackerOne signal was negative; banked queue was building

### Afternoon — the "go global" phase
- Scaled scans to 1,500-domain CORS sweep + 500-domain cloud-blobs + 1,000-domain takeover (paid corpus)
- Added 200-domain mega-corpus (Fortune 500, government, crypto, AI labs)
- 45 CORS hits on 100 security vendors + 85 hits on 200 mega-corpus

### Evening — the discovery
Live-verified the top candidates. Then Raffa flagged the suspicious pattern:

> "make sure that all the findings are relevant and really stand for an issue cause we do not want to waste our time or even worst being not proffesionals - its too common and it suspicious"

That gut-check triggered re-verification against REAL 200-OK endpoints (vs the 4xx error pages the scanner had matched on).

### The discovery
**Every single one of tonight's "verified" CORS findings turned out to be CDN error-page defaults, NOT real exploitable application-level misconfigs.**

| Target | 4xx flag | Real 200 endpoint | Result |
|---|---|---|---|
| Radware (Israeli sec vendor) | reflect+creds | NO CORS headers | FP |
| JFrog (Israeli, NASDAQ) | reflect+creds | acao=jfrog.com (PINNED), no creds | FP |
| Amazon.nl | reflect+creds | NO CORS headers | FP |
| Kroger | reflect+creds | partial CORS, no exploit | FP |
| OpenSea | reflect+creds | NO CORS on real API | FP |
| SpaceX | reflect+creds | reflect+creds on .well-known (public data) | Low-impact, not bountiable |

Submitting any of those would have damaged H1/Bugcrowd signal for 4+ weeks.

### The decision
- Lictor v0.2 patrol script CANNOT distinguish error-page CORS FPs from real bugs
- A reasoning agent (LLM) can — by comparing 4xx response headers with 200 response headers on the same host
- This requires a fundamentally different architecture: scan → probe → REASON → draft → review

That's v3. Born tonight.

## The 6 FP filters Owl now runs

1. **CloudFront x-amz-cf-id** (live-distro check)
2. **Azure CloudApp 4xx-only** (live signin pages with "404" in JS rejected)
3. **GitHub Pages org-exists** (api.github.com/users/{org} → 200 means real org)
4. **Netlify strict markers** (must contain "Not Found - Request ID:" OR "There is nothing here yet")
5. **Heroku canonical** (must contain "No such app" + status 404)
6. **CORS error-page-only** (NEW — compare 4xx vs 200 CORS, reject if 200 doesn't share the misconfig) — THE V3 SUPERPOWER

## Lessons codified in v3

1. **Cost asymmetry**: 1 bad submission = 4+ weeks of signal recovery. Bias REJECT.
2. **Verify impact, not headers**: scanner matches a header. Owl verifies the IMPACT by comparing with a 200 endpoint.
3. **Ethics footer mandatory**: "I did NOT do X" on every submission — never use leaked keys to verify them
4. **No money/star asks in disclosures**: spam-disguised-as-security damages the field
5. **Israeli targets prioritized** when there's a tie (cultural + geographic alignment)
6. **Workspace files are the protocol**: agents read/write files, no agent-to-agent calls

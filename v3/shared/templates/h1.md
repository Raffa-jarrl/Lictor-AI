# HackerOne submission template (Raven uses this)

## Form fields Raven must populate

| Field | Where it comes from |
|---|---|
| **Asset** | finding.program.scope_match (must exactly match a dropdown value) |
| **Weakness** | finding.class → H1 taxonomy mapping (see below) |
| **Severity** | finding.severity.cvss (CVSS:3.1 vector format) |
| **Title** | Raven generates ≤140 chars |
| **Description** | The body of this template, filled in |

## H1 Weakness taxonomy mapping

| Lictor finding class | H1 weakness |
|---|---|
| subdomain-takeover | Subdomain Takeover |
| cors-reflect-with-creds | Cross-Origin Resource Sharing (CORS) — Insecure Configuration |
| graphql-introspection | Information Disclosure |
| sourcemap | Information Disclosure |
| cloud-blob-exposure | Information Disclosure |
| exposed-file | Information Disclosure |
| credential-leak | Insufficient Anti-automation/Anti-CSRF (rare — usually disclosure, not bounty) |

## Description body template

```markdown
## Summary

{2-4 sentence summary of what was found, WHERE it was found, and WHY it's exploitable. No hype words.}

## Steps to reproduce

1. {first step — usually a curl or DNS command, with full syntax}

   {exact command, no placeholders}

2. {second step — the actual finding}

   {expected output / actual output}

3. {optional: a third confirmation step}

## Impact

{2-4 specific impact bullets. Each bullet starts with a concrete action an attacker could take. Avoid "could be devastating" — say WHAT specifically happens.}

- {specific impact 1}
- {specific impact 2}
- {specific impact 3 — bonus context if it makes the case stronger}

## Remediation

{Numbered list of specific fixes with config flags / code changes / commands.}

1. {fix 1 — specific}
2. {fix 2 — specific}
3. {long-term defense — audit recommendations}

## Tooling

Discovered via Lictor (https://lictor-ai.com) — open-source security scanner (Apache 2.0). {1-sentence summary of which Lictor scanner produced the finding and how.}

## Note on scope

Per the {program-name} H1 program scope, `{asset}` is listed with {key scope attributes, e.g. "eligible_for_bounty=true, max_severity=critical"}. The reported {hostname/endpoint} is a direct match for this scope item. {If there's a prior accepted report under this account, mention it — proves the program is open to this reporter.}

## References

- {Most-relevant standards link — W3C Fetch / OWASP CHEAT_SHEET / can-i-take-over-xyz / etc.}
- {Optional: another related CWE / OWASP link}

---

Ethical-disclosure note: I have NOT {specific verb of what I refrained from doing — "exploited the misconfig against real authenticated users" / "used the leaked credential" / "downloaded user data from the public bucket"}. The exploitation pattern is described to illustrate impact, not executed. All probes used neutral test origins (`attacker.example.com`) without authenticated sessions / credentials.
```
